# Plan テンプレート

> 計画ファイル（`.claude/addf/plans/`、ADDF 本体では `.claude/addf/plans-add/`）の標準テンプレート。
> 下の「標準テンプレート」をコピーして `<番号>-<slug>.md` として保存し、プレースホルダを埋める。
> まだ実装計画に固まっていない検討段階のメモには、末尾の「検討スタブ variant」を使う。
>
> 出典: ダウンストリームで独立起草された計画8本が同一構造に収束した実績（Plan 0033 項目4）。
> `## 実装状況:` ヘッダは lint（TODO ⇔ Plan 状態の突合）の保護対象に入るため必ず残すこと。
> PR を作成する際の本文書式（対象 Plan リンク・進捗位置欄）は `.claude/addf/guides/pr-format.md` を参照。
>
> ADDF 本体（`.claude/addf/plans-add/`）では既存 Plan の慣行（目的 / 項目・フェーズ / 影響範囲 /
> 完了条件 / 関連 Plan）を優先してよく、本テンプレートの追加見出し
> （現状の挙動・テスト方針・破壊的変更の許容範囲・要オーナー確認）は任意。

---

## 標準テンプレート

```markdown
# Plan <番号>: <日本語タイトル>

## 実装状況: 未着手

<!-- 大改造級（全域に触れ、投機・並走ブランチと両立しない計画 — ディレクトリ移動・
     一括リネーム等）は一発通し切りマーカーを付ける。該当する場合は**このコメントを削除して**、
     「## 実装状況:」ヘッダの直後に次の1行を置く（完成形 — 行頭に置き、先頭に空白を入れない）:

       execution_style: one-shot

     意味と実施様式（事前清算・リハーサル前置・単一セッション完走・失敗時は backup ref へ
     巻き戻し優先・オーナー同席推奨）の単一ソースは
     .claude/addf/guides/speculative-development.md「one-shot」。
     /addf-speculate の窓検出（手順 1.8）がこの行を行頭一致の grep で拾うため、
     プレースホルダや解説文の行頭には書かないこと（上の完成形例が字下げされているのもそのため）。
     TODO の状態注記にも「one-shot」を併記する。
     該当しない計画ではこのコメントごと削除してよい。 -->

> 出典: <この計画が生まれたきっかけ — オーナー指示・レビュー指摘・Issue 番号・分離元 Plan など>

## 関連 Plan

<!-- 派生・依存・分離の関連を markdown リンクで記す。
     リンクは双方向を原則とする（knowhow 記事の相互リンクと同じ双方向原則。
     片リンクは辿れない側で関連が失われる）— 関連 Plan 側にも本 Plan へのリンクを追記すること。
     関連がなければこのセクションごと削除してよい。 -->

- [Plan <番号>: <タイトル>](<ファイル名>.md) — <関係の一行説明（依存・派生・分離・対の仕組み等）>

## 目的

<何を達成したいか。1〜3行>

## 現状の挙動

<いま何がどう動いていて、何が問題か>

## 変更内容（項目・フェーズ）

<!-- 規模に応じて「項目1・項目2…」または「フェーズ1・フェーズ2…」で分割する。
     フェーズ分割した Plan は途中フェーズの PR がマージされても完了ではない —
     PR 本文の「計画の進捗位置」欄（.claude/addf/guides/pr-format.md）で残フェーズを可視化する。 -->

### 項目1: <見出し>

- **対象**: <変更するファイル・コンポーネント>
- <変更の内容>

## 影響範囲

<変更が波及する範囲 — ファイル・同期ペア（lint 対象）・ダウンストリーム配布への影響など>

## テスト方針

<どう検証するか — 既存テストへの追加・ドリフト注入・手動確認など>

## 破壊的変更の許容範囲

<後方互換を壊してよい範囲。なければ「なし」>

## 要オーナー確認

<オーナーの判断が必要な事項。なければ「なし」>

## 完了条件

- [ ] <機械検証できる条件（テスト・lint の通過など）を優先して書く>
- [ ] <人間判断が必要な条件は `<!-- human-judgment -->` マーカーで明示する>

## AI 実装時間見積もり

<任意セクション。書く場合は「1セッション以内」等の AI 実装時間で書く（人間換算の期間見積もりは不要）>
```

## 検討スタブ variant（実装計画に固まる前のメモ）

計画として詰め切れていない着想を捨てずに置いておくための簡略書式。
着手のトリガーが満たされたら標準テンプレートに書き直して着手する。
検討スタブも標準 Plan と同じディレクトリ・同じ番号列に置いてよい（番号を消費する）。
実装計画に昇格したら同番号のまま標準構造に書き換える。

```markdown
# Plan <番号>: <日本語タイトル>（検討スタブ）

## 実装状況: 未着手

## 分かっていること

- <確度の高い事実・すでに決まっていること>

## 未解決の問い

- <決めないと着手できないこと・オーナーに聞くべきこと>

## 着手のトリガー

- <どうなったらこのスタブを標準テンプレートに書き直して着手するか>
```
