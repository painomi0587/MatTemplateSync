# 陳腐化しやすい knowhow 記述パターン

Plan 0032 の🟡棚卸しで、7件中3件の訂正が同じ「陳腐化の型」に嵌まっていた。
今後 knowhow を書くとき／書き直すときにこの型を避ける・逃がす。

## 発見した知見

### 3つの陳腐化パターン

| # | パターン | 症状 | 事例 |
|---|---|---|---|
| 1 | **未実装・改善余地の羅列** | 実装完了時に訂正が必要になる | `skill-design-patterns.md` の「改善の余地: 計測フックの導入」→ `skill-usage-log.sh` 配線済みで訂正 |
| 2 | **例示リストの現物埋め込み** | 実物（settings.json / .gitignore 等）がドリフトすると乖離する | `permission-settings-pattern.md` の settings.json 例が `mktemp` 等を欠いていた |
| 3 | **判定分岐図の閉じた列挙** | スキル本体に新ケースが増えると分岐図が古くなる | `existing-project-install-pattern.md` の 4分岐に「部分導入」が抜けていた |

### 逃がし方（書くときの作法）

- **パターン1（未実装リスト）**: 「未実装」ではなく「原理・パターン・トレードオフ」を書く。
  実装完了しても記述が古くならない。どうしても未実装リストを書くなら **~~取消線~~ で
  実装済みマークを残す**（消さない — 変遷を歴史として保管）
- **パターン2（現物埋め込み）**: 例示は「基準の説明」に留め、詳細列挙は現物ファイルに
  委譲する。「詳細は `.claude/settings.json` を参照」と1文書く。既に `sync-lint-design.md` の
  「列挙を持たない単一ソース化」原則の応用
- **パターン3（分岐図）**: 分岐図を書くなら、その分岐を実装するスキル本体側の分岐（e.g.
  `addf-init.md` Phase 1）と**ペアで同期する仕組み**が理想（未整備）。それまでは:
  - 分岐図に「詳細は `.claude/commands/xxx.md` を参照」を添える
  - `last_verified` を更新するたびに分岐図の網羅性も点検する（意思で覚えず、
    revise 時のチェック観点として運用に落とす）

## プロジェクトへの適用

### 新規 knowhow 作成時の自己チェック

- 「未実装」「改善の余地」の節を書いたか？ → 書いたら実装完了時に訂正することを覚悟する
- 現物リスト（設定値・ファイル名列挙）を書いたか？ → 「詳細は現物を参照」で逃げられないか検討する
- 分岐図・判定表を書いたか？ → その分岐を実装するファイルへのリンクを添える

### revise 時のチェック観点

- 過去に列挙した「未実装」項目が実装されていないか（→ 取消線で歴史保管）
- 例示した現物ファイルが今もその形か（→ 差分を訂正）
- 分岐図の網羅性（新ケースが追加されていないか）

## 注意点・制約

- パターン1〜3を完全に避けようとすると knowhow が抽象化しすぎて具体性を失う。
  「具体性を保ちつつ、訂正が要ると気づける形」に留めるのが実務のバランス
- 「機械的同期で守る」対象になれば理想だが、現時点で本記事の3パターンは lint 化されていない
  （sync-lint-design のペア候補ではある — スキル本体の分岐 ↔ knowhow の分岐図）。将来的な lint 化余地
