# ADDF Plan Audit — 未完了埋没タスクの掘り起こし

「完了扱いになっているが未完了タスクが残っている計画」を走査・検出し、
処置の3択をオーナーに提案する。フェーズ分割 Plan の途中マージ・レビュー指摘の
「Low は Plan に記録」・番号だけの後続 Plan 言及などが年月とともに埋没するのを、
一回の棚卸しで視界に戻す。

## 走査対象 — upstream/downstream の切り替え

| プロジェクト種別 | Plan ディレクトリ | 突合する TODO |
|---|---|---|
| ADDF 本体（`CLAUDE.repo.md` に「ADDF 開発プロジェクト」宣言） | `.claude/addf/plans-add/`（`.claude/addf/plans/` も存在すれば対象） | `.claude/addf/plans-add/TODO.addf.md` |
| ダウンストリーム（「ADDF 利用プロジェクト」宣言、または宣言なしで `.claude/addf/lock.json` が存在） | `.claude/addf/plans/` | `TODO.md` |

種別判定は明示シグナル（一次根拠: `CLAUDE.repo.md` の種別宣言 / フォールバック:
`addf-lock.json` の存在）で行い、ファイルの物理存在では判定しない（存在≠所有 —
lint-template-sync と同じ規約）。

## 役割分担 — 常設 lint との補完関係

| 仕組み | 見えるもの | 実行タイミング |
|---|---|---|
| lint-plan-status（`/addf-lint` セクション12） | 構造化された矛盾（完了条件チェックボックス未完 × ヘッダ「完了」） | 常時（新規の嘘をブロック） |
| 本スキル | 意味的な埋没（「先送り」「残課題は別途」等の文章、番号ずれ、TODO との不整合） | 一回きりの掘り起こし＋任意の棚卸し |

検出は決定的スクリプト・解釈はエージェントの役割分担（`.claude/addf/knowhow/ADDF/sync-lint-design.md`）。
層1・層3の機械検出はスクリプトの実行結果を取り込み（**再実装しない**）、
層2の意味読解だけを本スキル＝エージェントが担う。PlanTemplate 準拠の新運用
（チェックボックス完了条件＋常設 lint）が回り始めた後は、本スキルの定常実行は不要
（任意の棚卸しとして残る）。

## 手順

（各コマンドはリポジトリルートで実行する）

### 1. 層1: 構造検査（スクリプト実行）

```bash
uv run --python 3.11 .claude/addf/addfTools/lint-plan-status.py
```

（uv が無い環境では `python3` で直接実行できる — stdlib のみ）

- ERROR（完了ヘッダ × 未チェック残存）と WARNING（表記ゆれ状態ヘッダ）を検出一覧に取り込む
- **SKIP された旧書式 Plan（チェックボックス非採用）は lint の死角** — ファイル名列挙が
  出力されるので、層2の読解対象として必ず引き継ぐ

### 2. 層2: 意味的パターン（エージェントの読解）

読解対象は2系統で決める:

**(a) 層1 の SKIP 一覧 — 全件読解**: 層1で SKIP された旧書式 Plan は、下記 grep の
ヒット有無にかかわらず**全件**開いて後述の判定基準を当てる（SKIP は lint の死角
そのものなので grep で間引かない。報告書式の「SKIP n（旧書式 → 層2 で全件読解済み）」は
この読解を指す — 全件読んで初めてこの行が真になる）

**(b) SKIP 以外の Plan — grep で候補収集**: あたりを付ける（これは読む場所の
絞り込みであって機械判定ではない）:

```bash
grep -nE '先送り|残課題|別途|独立計画|可能性あり|検討する|推奨|保留|指示待ち|承認待ち|申し送り|残り[はの:：]|フェーズ[0-9A-Za-z]+以降' .claude/addf/plans-add/*.md
```

（ダウンストリームでは `.claude/addf/plans/*.md`。語彙は固定リストではない — Plan の語調に
合わせて追加してよい。取りこぼしより過剰ヒットを許容し、判定は本文読解で行う）

> シェル挙動の注記: ヒット0件のとき grep は exit 1 を返すが、これは異常ではない
> （検出なしの正常終了）。また対象ディレクトリに `.md` が1つも無い場合はシェルによって
> 挙動が異なる（fish は glob 不一致でコマンド自体がエラーになる等）— Plan ファイルが
> 無いなら層2は「対象なし」として先へ進んでよい

ヒットした Plan は行単位ではなく**全文**を読み、文中で言及される後続 Plan・TODO の
該当箇所も併せて開く。そのうえで以下を判定する:

- **後続 Plan の実在と相互リンク** — 「独立計画で対応」「Plan 00NN 推奨」等の文言が指す
  回収先が実在するか。番号だけの言及は実 Plan と突き合わせる（**番号ずれ** — 言及時点の
  予定番号が後から別の Plan に取られている埋没 — に注意）。実在してもリンクが片方向なら
  相互リンク欠如として拾う（双方向原則は `.claude/addf/guides/pr-format.md` の相互リンク規約参照）
- **関連 Plan セクションの風化** — 未来形・条件文言（「〜する想定」「〜の可能性あり」
  「実装時に検討」）が、参照先 Plan の実装状況ヘッダ（完了済み等）と食い違ったまま
  残っていないか（Plan 0035 フェーズA で導入初日に実測された型）
- **ヘッダ注記と本文の食い違い** — 実装状況ヘッダの注記（「残りは〜のみ」等）と
  本文・完了条件セクションの実態が一致しているか

検出項目には**該当行番号と引用テキスト**を必ず記録する（後から該当箇所を再訪できる
ように。報告書式・監査記録の表の列と対応させる。行番号は走査時点のものと注記する）

### 3. 層3: TODO との突合

```bash
uv run --python 3.11 .claude/addf/addfTools/lint-template-sync.py
```

ペア6（TODO ⇔ Plan 実装状況ヘッダ）の結果を検出一覧に取り込む。
さらに**ペア6が見ない粒度**をエージェントが読む:

- 状態注記の食い違い（例: Plan ヘッダは「完了（○○のみ承認待ち）」なのに TODO 側は
  素の「完了」— 保留中の承認が TODO から見えない）
- TODO の「現在のフェーズ」行と各行の状態の食い違い
- アーカイブ済み行・削除済み Plan への参照の残存

### 4. 検出結果の報告と処置の提案

検出項目ごとに、処置3択のいずれか（複数可）を**提案**として付す
（いずれも提案であり、エージェントが直接実行する指示ではない）:

1. **回収計画に起こすことを提案する** — 未完了項目を集約した新 Plan を起草して
   TODO に登録する（Plan 0026 型。出典 Plan と相互リンクする — 双方向原則）。
   目安: 残作業が独立起票に値する分量のとき
2. **ヘッダを訂正することを提案する** — 実態が「一部完了」なら Plan ヘッダと TODO を
   「一部完了（残り: …）」へ訂正する。
   目安: 実作業は不要で、記述の言い回し・参照番号だけが実態とずれているとき
3. **意図的な完了と明記して閉じることを提案する** — 残課題が「対応しない」と
   判断済みなら、その旨を Plan に明記して完了のまま閉じる。
   目安: 対応しない（または条件が満ちたら起票する）判断が既に下っているとき

### 5. 検出一覧の永続化（チャットの外に残す）

検出一覧はチャット報告に加えて、**必ずリポジトリ内へ書き残す**（掘り起こした結果
自体がチャットの外で埋没しては本末転倒。オーナーが後から判断できる材料を残す）:

- **トリガーとなった Plan がある場合**（ADDF 本体で特定 Plan を起点に実行したとき・
  addf-migrate のワンショット案内で実行したとき等）: その Plan の実装記録セクションに
  検出一覧表（Plan / 該当行 / 内容の引用 / 処置の提案＋理由1行）を追記する
- **定期棚卸しとして実行した場合**: `.claude/addf/plans-add/`（ダウンストリームでは
  `.claude/addf/plans/`）に監査記録として Plan ファイルを新設し（例:
  `NNNN-plan-audit-YYYY-MM-DD.md`）、同じ表を記載して TODO に登録する。
  同月内の再実行なら既存の監査記録 Plan への追記でよい

- 処置の最終判断はオーナーが行う — 各検出に付した処置提案の採否・完了状態の変更は
  オーナーが確認して判断する <!-- human-judgment -->（エージェントは提案と永続化まで
  で止まり、既存 Plan の完了状態を単独で書き換えない）

## 報告書式

検出の有無にかかわらず以下の形式で報告する。**検出 0 件でも「0件 — 3層とも整合」を
明記して報告する**（無言終了しない — 走査した事実そのものが棚卸しの成果）。

```
╔══════════════════════════════════════╗
║  ADDF Plan Audit Results             ║
╚══════════════════════════════════════╝

対象: .claude/addf/plans-add/ (N Plan) / TODO: .claude/addf/plans-add/TODO.addf.md

層1 構造検査:   ERROR n / WARNING n / SKIP n（旧書式 → 層2 で全件読解済み）
層2 意味読解:   検出 n 件（SKIP m 件を全件読解 + grep 候補 k Plan を読解）
層3 TODO 突合:  検出 n 件

## 検出一覧（処置提案付き。行番号は走査時点）

| # | Plan | 層 | 該当行 | 内容（引用） | 処置の提案 |
|---|---|---|---|---|---|
| 1 | 0012 | 1 | L3 | ヘッダ「完了」× 完了条件 `- [ ]` 残存2件 | ヘッダを「一部完了（残り: …）」へ訂正することを提案 |
| 2 | 0026 | 2 | L26 | 「独立 Plan 0027 で対応推奨」の 0027 が別 Plan に採番済み（番号ずれ） | 回収計画に起こすことを提案 |
| 3 | 0031 | 3 | — | Plan ヘッダ「完了（承認待ち）」× TODO は素の「完了」（保留が TODO から見えない） | TODO の状態注記を訂正することを提案 |

検出 0 件の場合: 「検出 0 件 — 3層とも整合」

永続化先: <手順5 で書き残した Plan / 監査記録ファイルのパス>
```

## 経験の活用

- 実行前に `addf-plan-audit.exp.md` が存在すれば読み、過去の経験を考慮する
- 実行後、新たな教訓（誤検出しやすい文言・見逃した埋没の型など）があれば
  `addf-plan-audit.exp.md` に追記する
